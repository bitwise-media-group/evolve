## What CodeQL reported

The query flagged the capacity argument of a `make` and traced one operand back to a "potentially large value" — the
JSON produced from a decoded spec file at
[`internal/encfmt/encfmt.go:55`](https://github.com/bitwise-media-group/evolve/blob/325d0fda592bc403e64532e722f163a5ee25b4cf/internal/encfmt/encfmt.go#L55):

```go
merged := make([]Assertion, 0, len(e.Expectations)+len(e.Assertions))
```

This is the
[`(*EvalsFile).normalize`](https://github.com/bitwise-media-group/evolve/blob/325d0fda592bc403e64532e722f163a5ee25b4cf/internal/evalspec/evals.go#L162)
merge step: an eval's authored `expectations` are expanded into leading `llm` assertions, and the slice is pre-sized to
hold both groups before they are appended. This finding flags the first operand of the `+` (`len(e.Expectations)`);
[finding 4](https://github.com/bitwise-media-group/evolve/security/code-scanning/4) flags the second
(`len(e.Assertions)`) — see [4.md](./4.md). The analysis below covers both.

## This is a re-detection of findings 1 and 2

This is the **same allocation** that was triaged as a false positive in findings
[1](https://github.com/bitwise-media-group/evolve/security/code-scanning/1) and
[2](https://github.com/bitwise-media-group/evolve/security/code-scanning/2) (see [1.md](./1.md), [2.md](./2.md)). The
`normalize` method, and this `make` line with it, moved from
[`internal/evalspec/evalspec.go:188`](https://github.com/bitwise-media-group/evolve/blob/73a49b31f27465a11cf65a99c23ae0651396cde8/internal/evalspec/evalspec.go#L188)
(its location when findings 1 and 2 were opened) to
[`internal/evalspec/evals.go:166`](https://github.com/bitwise-media-group/evolve/blob/325d0fda592bc403e64532e722f163a5ee25b4cf/internal/evalspec/evals.go#L166)
in the `refactor(evalspec): split triggers and evals into separate files` commit, which split the package's single
source file into `evals.go` and `triggers.go`. CodeQL closed findings 1 and 2 as `fixed` when the line disappeared from
its old location and opened findings 3 and 4 against the byte-identical code in its new home. No logic changed — only
the file and line.

## Why this is a false positive

**1. Both addends are lengths of slices already materialized in memory.**

`e.Expectations` (`[]string`) and `e.Assertions` (`[]Assertion`) are populated by `json.Unmarshal` inside
`encfmt.DecodeFile` _before_ line 166 runs. The `make` does not allocate from an attacker-chosen count — it pre-sizes
the concatenation of two slices that already coexist in memory, and the next three lines append exactly those elements
into `merged`. The capacity is by construction no larger than the number of elements the process already holds.

**2. The sum cannot overflow on any supported platform.**

evolve ships only 64-bit builds (linux, darwin, windows × amd64, arm64), where Go's `int` is 64-bit. For
`len(e.Expectations) + len(e.Assertions)` to reach 2^63 the process would already hold ~9.2×10¹⁸ elements in memory —
each a 16-byte string header or a multi-field `Assertion` struct, i.e. tens of exabytes of resident RAM. The inputs
cannot be decoded in the first place, so the addition that consumes their lengths cannot wrap.

**3. The taint premise does not hold.**

The "large value" CodeQL traces from
[`encfmt.go:55`](https://github.com/bitwise-media-group/evolve/blob/325d0fda592bc403e64532e722f163a5ee25b4cf/internal/encfmt/encfmt.go#L55)
is the decoded file content, whose size is bounded by the file already read into memory and unmarshaled. There is no
multiplication and no from-scratch allocation sized by an attacker-controlled count — only `len()` of data that
demonstrably already exists. The pattern the rule warns about (an untrusted size flowing into an unguarded allocation)
is absent.

## Resolution

No code change is warranted: the construct is a standard capacity hint for a slice merge and cannot overflow on any
platform evolve targets, and the disposition is unchanged from findings 1 and 2. Because CodeQL is run over first-party
`internal/` source with no path filter — by design, so our own code is analysed — there is no path-ignore escape and
none is wanted. Both findings (3 and 4) flag the two operands of this single expression and are dismissed in the
code-scanning UI as **false positive**.

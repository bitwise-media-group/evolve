## What CodeQL reported

This is the same allocation as finding [3](https://github.com/bitwise-media-group/evolve/security/code-scanning/3), on
the same line:

```go
merged := make([]Assertion, 0, len(e.Expectations)+len(e.Assertions))
```

CodeQL emits one alert per operand of the `+`: finding
[3](https://github.com/bitwise-media-group/evolve/security/code-scanning/3) flags `len(e.Expectations)` (cols 35–54) and
this finding flags `len(e.Assertions)` (cols 55–72). The "potentially large value" is traced to the JSON produced from a
decoded spec file at
[`internal/encfmt/encfmt.go:55`](https://github.com/bitwise-media-group/evolve/blob/325d0fda592bc403e64532e722f163a5ee25b4cf/internal/encfmt/encfmt.go#L55)
in both.

Like finding 3, this alert is a re-detection of the false positive previously triaged as findings
[1](https://github.com/bitwise-media-group/evolve/security/code-scanning/1) and
[2](https://github.com/bitwise-media-group/evolve/security/code-scanning/2). The `make` line moved from

(its location when findings 1 and 2 were opened) to
[`internal/evalspec/evals.go:166`](https://github.com/bitwise-media-group/evolve/blob/325d0fda592bc403e64532e722f163a5ee25b4cf/internal/evalspec/evals.go#L166)
in the `refactor(evalspec): split triggers and evals into separate files` commit; CodeQL closed the old alerts as
`fixed` and opened findings 3 and 4 against the byte-identical code. See [3.md](./3.md) and [1.md](./1.md) for the full
lineage.

## Why this is a false positive

The disposition is identical to finding 3 — the full reasoning is in [3.md](./3.md). In short: both `e.Expectations` and
`e.Assertions` are slices `json.Unmarshal` has already materialized in memory before line 166, so the `make` merely
pre-sizes the concatenation of two existing slices rather than allocating from an attacker-chosen count. On evolve's
64-bit-only build targets the sum of two in-memory slice lengths cannot reach 2^63 without tens of exabytes of resident
RAM, so the addition cannot overflow.

## Resolution

No code change is warranted; see [3.md](./3.md) for the full analysis. This finding is dismissed in the code-scanning UI
as **false positive** alongside finding 3.
